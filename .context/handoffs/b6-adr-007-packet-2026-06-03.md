## TL;DR

ADR-007 must capture **six** things for the Clay scheduler layer to be auditable, bisectable, and operator-readable:

1. **Lifespan is a thin shell.** It stamps `app.state`, logs, and delegates to `ClayScheduler.start()` / `shutdown()`. No audit writes; no registry mutations; no surprises.
2. **ClayScheduler is the audit chokepoint.** All scheduler-layer audit verbs (`scheduler.started`, `scheduler.stopped`, `service.status_changed` on tick exceptions) are written from inside `ClayScheduler.start()` / `shutdown()` / `_handle_job_error()` and the jobs themselves. The lifespan body never writes audit events.
3. **Per-tick side-effects are transition-only** (B3b anti-flood pattern). Steady-state ticks write zero audit records; transitions write exactly one. The `_failing` episode flag (B4 #11) resets on the next successful tick so a fresh failing episode re-emits.
4. **Jobs must be side-effect-free at the service-method level** (B0 lesson, codified by `obs-2026-06-02-002`). If a service method writes audit/bus on every call, it must grow an `emit: bool = True` parameter before it can be wrapped in a scheduler job (B4/B5 refactor pattern).
5. **Async vs sync routing is determined by the registered callable's signature**, not by the wrapper internals. `health-tick` + `reliability-recheck` are sync (`_run_safely` → ThreadPoolExecutor); `ingestion-cycle` is async (`_arun_safely` → event loop). Two wrappers + shared error-helper, NOT one `inspect.isawaitable` wrapper (that would silently regress B3b/B4).
6. **`scheduler.started.jobs` payload reflects truth, not intent.** Built from `apscheduler.get_job(id) is not None` lookups (Q2 single source of truth). A misconfigured flag + missing dep path produces a payload that does not lie about the missing job.

The remaining sections of this packet are the **detailed contract** — env-gate surface, audit ordering, partial-failure matrix, anti-flood patterns, single-worker assumption, the routing matrix, and the B6 test that pins each one.

## 1. Lifespan contract (the thin shell)

`backend/src/clay/api/lifespan.py` is a FastAPI `@asynccontextmanager async def lifespan(app)`. Its sole responsibilities are:

1. **Stamp `app.state`** with two references that outlive any single request:
   - `app.state.started_at: datetime | None` — diagnostic marker.
   - `app.state.scheduler: ClayScheduler | None` — the live scheduler instance, or `None` if `CLAY_SCHEDULER_ENABLED=false`.
2. **Gate on `scheduler_settings.enabled`** (env `CLAY_SCHEDULER_ENABLED`).
3. **Delegate to `ClayScheduler.start()` / `shutdown(wait=True)`** — those methods own all audit writes and registry transitions.
4. **Log** at info level on entry/exit (no audit).

**Confirmed by** `lifespan.py:70-111` + the B6 test `test_session_scheduler_state_walk_stopped_healthy_stopped` (`tests/integration/test_scheduler_lifespan.py:131`).

### 1.1 Boot order (verified by B6)

```python
# api/lifespan.py:80-110
app.state.scheduler = None           # 80  ← guard
app.state.started_at = None          # 81  ← guard
logger.info("clay.api.lifespan: startup")  # 82  ← log only
try:
    app.state.started_at = datetime.now(UTC)  # 84  ← stamp
    if scheduler_settings.enabled:            # 85  ← env gate
        scheduler = ClayScheduler(...)         # 86-95  ← DI
        scheduler.start()                       # 96  ← audit "scheduler.started" happens INSIDE
        app.state.scheduler = scheduler         # 97  ← ONLY after start() succeeded
        ...
    else:
        app.state.scheduler = None              # 100
    yield                                        # 105
finally:
    if app.state.scheduler is not None:         # 107
        app.state.scheduler.shutdown(wait=True)  # 108  ← audit "scheduler.stopped" INSIDE
    ...
    app.state.scheduler = None                   # 110
    app.state.started_at = None                  # 111
```

**Confirm (a) — order of `scheduler.start()` vs `app.state.scheduler = scheduler`** (B6 verify): **`scheduler.start()` runs BEFORE `app.state.scheduler = scheduler`**. If `start()` raises, `app.state.scheduler` remains at the guard value `None`. This is the invariant pinned by `test_startup_failure_keeps_state_clean` (`tests/integration/test_scheduler_lifespan.py:368`).

**Confirm (b) — who writes `service.status_changed` for `control-api`?** (B6 verify): **`registry.update_status()` is a pure mutation** (`services/registry.py:32-40` — `record.status = status; record.last_error = error; return record`); it does NOT call `audit_writer.write` internally. The `bootstrap.py:148` call `registry.update_status("control-api", ServiceStatus.HEALTHY)` writes nothing to the audit log. Audit `service.status_changed` is written by **call-sites only**, in:

- `scheduler/service.py:_handle_job_error` (line 456-464) — per-tick exception.
- `scheduler/jobs.py:HealthTickJob.run` (line 179) — per-tick service status change.

This means there is currently an **inconsistency** in the audit topology: the per-tick exception path writes `service.status_changed`; the normal startup/shutdown transition path does NOT (e.g. `control-api STOPPED→HEALTHY` at import, `session-scheduler STOPPED→HEALTHY` at `start()`). **Open ADR-007 question:** should `registry.update_status(...)` optionally emit `service.status_changed` on every transition, or should the per-tick sites continue to be the only writers?

### 2.3 Per-tick audit (the B3b anti-flood pattern)

| Job | Per-tick audit | Per-tick bus | Anti-flood mechanism |
|-----|----------------|--------------|---------------------|
| `health-tick` (B3b) | `service.status_changed` only on **transition** (any service's status change, with pre-tick capture so re-asserts of `HEALTHY` don't fire). | `health.tick` every tick. | Pre-tick `pre_status` capture + transition diff. |
| `reliability-recheck` (B4) | `reliability.rechecked` only on **transition** (3-tuple `(release_readiness_status, blocking_gate_count, warning_gate_count)` diff). | `reliability.updated` on the same transition. | First-run seed (`_cache: tuple \| None = None` → seed, no emit) + 3-tuple diff. |
| `ingestion-cycle` (B5) | `ingestion.run` only on **transition** (2-tuple `(bool(incidents), freshness_state_transitions)` diff). | same. | Same pattern. **Поправка 2 (Emma):** `freshness_state_transitions` is incremented only on `INSERT` or state-change `UPDATE` of `MarketFreshnessStatus`, NOT on timestamp-only touches. Steady = `(0, 0)` → no emit. |

**Per-error audit (B4 #11 isolated error policy):**

| Job | On `on_error(exc)` |
|-----|---------------------|
| `health-tick` | **Default B3b path** — `session-scheduler` → `ERROR` + `service.status_changed` audit on first transition. No re-raise; APScheduler keeps the slot alive. |
| `reliability-recheck` | **Isolated path** — `reliability.recheck_failed` once per failing episode. `_failing = True`; reset to `False` on the next successful tick. `session-scheduler` is NOT mutated. No re-raise. |
| `ingestion-cycle` | **Isolated path** — `ingestion.cycle_failed` once per failing episode. Same `_failing` + reset semantics. Same isolation. |

**Error-policy routing** is parameterized at the wrapper level: `ClayScheduler._run_safely(..., on_error=job.on_error)` and `_arun_safely(..., on_error=job.on_error)`. The wrapper delegates to `_handle_job_error` (single source, no duplication) which either runs the custom `on_error` (B4/B5 jobs) or the default B3b path.

**Confirmed by** the B5 unit tests in `tests/scheduler/test_clay_scheduler.py` and the B3b acceptance in `test_health_tick_job.py`. The B6 suite asserts the **wrapper-level** behaviour (jobs registered with the right executor, audit ordering at startup/shutdown); the per-tick transition logic is unit-tested at the job level (B3b/B4/B5 unit tests, not duplicated in B6).

## 4. Job registration matrix (verified by B6)

| Job id | Slice | Registered callable | APScheduler executor | Runs in | Anti-flood | First-run seed |
|--------|-------|---------------------|----------------------|---------|-----------|----------------|
| `health-tick` | B3b | `ClayScheduler._run_safely` (sync) | `"default"` → `ThreadPoolExecutor(max_workers=4)` | ThreadPool | pre-tick capture of `session-scheduler` status, transition diff on any service | n/a (per-tick) |
| `reliability-recheck` | B4 | `ClayScheduler._run_safely` (sync) | `"default"` | ThreadPool | `_cache: tuple \| None = None` + 3-tuple `(release_readiness_status, blocking_gate_count, warning_gate_count)` diff; `_failing` episode flag | seed on first run, no emit |
| `ingestion-cycle` | B5 | `ClayScheduler._arun_safely` (async) | default → `AsyncIOScheduler` event loop | Event loop | `_cache` + 2-tuple `(bool(incidents), freshness_state_transitions)` diff; `_failing` episode flag | seed on first run, no emit |

**All jobs** are registered with `max_instances=1`, `coalesce=True`, `replace_existing=True`.

**Confirmed by** `test_jobs_registered_all_three` + `test_routing_matrix_sync_vs_async` (B6, `tests/integration/test_scheduler_lifespan.py:79` and `:108`). The routing matrix test inspects `apscheduler.get_job(id).func` and asserts via `inspect.iscoroutinefunction(...)`:

- `health-tick.func` and `reliability-recheck.func` are **not** coroutine functions (sync `_run_safely`).
- `ingestion-cycle.func` **is** a coroutine function (async `_arun_safely`).

This is the integration-level confirmation of Emma's B5 fragment D mandatory code fix — a sync wrapper around an `async def` would silently never await the coroutine, leaving ingestion dead.

**Why two wrappers, not one `async def` wrapper with `inspect.isawaitable`:** the registered callable's signature (`sync def` vs `async def`) determines APScheduler's routing — the wrapper's own internals do not. A single `async def` wrapper would silently move B3b/B4 sync DB-I/O onto the event loop (regression: freezes request handlers). Two wrappers + shared `_handle_job_error` = correct routing + no duplication. Documented at `scheduler/service.py:362-407`.

## 5. Side-effect-free precondition (the B0 lesson, codified)

`obs-2026-06-02-002` (B4 recon) established the rule: **before scheduling any service method, verify it's side-effect-free. If not, refactor with an `emit` flag or pure-compute split, then schedule.**

### 5.1 The pattern: `emit: bool = True` parameter

B4 refactor: `ReliabilityService.recheck(session, *, emit: bool = True)`.

```python
# reliability/service.py (B4)
def recheck(self, session, *, emit: bool = True) -> ReliabilitySnapshot:
    self._last_rechecked_at = datetime.now(UTC)
    if self.session_factory is not None:
        ReliabilityStateRepository(session).save(last_rechecked_at=self._last_rechecked_at)
    snapshot = self.build_snapshot(session)
    if emit:
        self.emit_recheck_events(snapshot)
    return snapshot

def emit_recheck_events(self, snapshot: ReliabilitySnapshot) -> None:  # public
    self._audit_writer.write("reliability.rechecked", {...})
    self._event_bus.publish("reliability.updated", {...})
```

B5 refactor: `IngestionCycleService.run_once(session, *, emit: bool = True)`. Same shape.

```python
# ingestion/service.py:126-151 (B5)
async def run_once(self, session, *, emit: bool = True) -> IngestionRunSummary:
    if self._lock.locked():
        raise IngestionCycleBusy("ingestion cycle already running")
    async with self._lock:
        summary = await self._do_run_once(session)
    if emit:
        self.emit_cycle_events(summary)
    return summary
```

**Pattern invariants:**

- `emit=True` is the manual-route default — full audit + bus, no behaviour change.
- `emit=False` is the scheduler-driven path — DB writes mandatory, audit + bus skipped.
- A public `emit_*_events(summary)` method is the single source of the payload shape, shared by the manual route (`emit=True`) and the scheduler-job (transition-only on top of `emit=False`).
- Anti-drift: one payload shape, two call sites.

**Confirmed by** the B4 unit tests in `tests/reliability/test_reliability_service.py` and the B5 unit tests in `tests/ingestion/test_ingestion_cycle_service.py` (not duplicated in B6 — the per-tick transition logic is unit-tested; B6 only tests the wiring).

### 5.2 The manual-route-vs-scheduler amplification

| Service | Manual call rate | Scheduler call rate | Audit/bus per call (without `emit` refactor) | Audit/bus per call (after `emit=False` + transition-only) |
|---------|------------------|---------------------|---------------------------------------------|--------------------------------------------------------|
| `reliability_service.recheck` | ~1/day | 288/day (5min) | 288 + 288 = 576/day | ~10-50/day (transitions only) |
| `ingestion_cycle_service.run_once` | ~1/day | 1440/day (60s) | 1440 + 1440 = 2880/day | ~10-50/day (transitions only) |

**Without** the `emit` refactor, Wave B would generate ~3400 audit events per day in the steady state, all noise. **With** the refactor + transition-only emit, Wave B generates ~10-50 audit events per day (one per real state change). This is the B3b anti-flood correctness.

## 7. Single-worker assumption (operational constraint)

`ClayScheduler` runs **in-process** on the asyncio loop. Multi-worker deployments (e.g. `uvicorn --workers N`) would start `N` `ClayScheduler` instances, each registering the same jobs at the same intervals, each writing their own audit records. The operator must run Clay single-worker or accept that background jobs run multiple times.

Out of scope for Wave B (would need leader-election via Postgres advisory lock, Redis, or similar). Documented at `scheduler/service.py:43-47`.

**Implication for B6:** the B6 integration tests are single-process by construction (`uv run pytest` runs in a single process). No leader-election concerns at the test level.

## 8. Lifespan side-effect boundary (the explicit guarantee)

For the ADR's "lifespan is side-effect-free" guarantee, the boundary is:

- **Inside the lifespan body** (lifespan.py:70-110): log lines + `app.state` mutations + delegation to `ClayScheduler.start()` / `shutdown(wait=True)`. **Zero audit writes.**
- **Inside `ClayScheduler.start()` / `shutdown()`** (`scheduler/service.py:148-187`, `:470-482`): audit writes for `scheduler.started` / `scheduler.stopped` + `registry.update_status(...)` (no audit, pure mutation). All scheduler-layer audit verbs originate here.
- **Inside the per-tick sites** (`scheduler/jobs.py:HealthTickJob.run`, `ReliabilityRecheckJob.run`, `IngestionCycleJob.run` + their `on_error` methods): audit + bus on transition only. Per-tick transition logic, anti-flood, first-run seed.
- **Inside the `_handle_job_error` helper** (`scheduler/service.py:414-468`): per-exception audit + registry mutation. The single source of the B3b default error policy (when `on_error=None`) and the B4/B5 isolated error policies (when `on_error=job.on_error`).

## 10. Open questions for ADR-007 (your call)

1. **Should `registry.update_status(...)` emit `service.status_changed` on every transition?** Currently it's a pure mutation; `service.status_changed` is written only by `ClayScheduler._handle_job_error` (per-tick exception) and `HealthTickJob.run` (per-tick service status change). The per-state-transition path (`control-api STOPPED→HEALTHY` at bootstrap, `session-scheduler STOPPED→HEALTHY → STOPPING → STOPPED` at lifespan) is **silent**. Three options:
   - (a) Keep current: call-sites must explicitly audit. Pro: no surprise audit. Con: two paths to remember.
   - (b) Move audit into `registry.update_status(...)` (optional flag, default off?). Pro: one source of truth. Con: every call site gets an audit, may surprise.
   - (c) Add a separate `registry.update_status_and_audit(...)` method. Pro: explicit, no default surprise. Con: two methods, more code.

2. **Should `ClayScheduler.shutdown()` failure leave `app.state.scheduler = None` (B6 #13 quirk)?** Currently: `apscheduler.shutdown` raise → `finally: app.state.scheduler = None` unreachable → `app.state` retains a reference to a dead `ClayScheduler`. Fix candidate: wrap `ClayScheduler.shutdown(wait=True)` in its own `try/except` inside the `finally:`, log the error, and unconditionally run `app.state.scheduler = None`. Decide: (a) fix in this ADR, (b) document the quirk and defer, (c) something else.

3. **Should `double_startup_does_not_crash` be fixed?** B3a soft debt #10: calling `ClayScheduler.start()` twice produces a duplicate `scheduler.started` audit. Documented as not-a-blocker. Decide: (a) harden `start()` to be idempotent, (b) leave the soft debt, (c) raise explicitly on double-start.

4. **Should `apscheduler.AsyncIOScheduler` + `ThreadPoolExecutor(max_workers=4)` be the only allowed topology?** Out of scope for Wave B but worth deciding for the ADR. Alternative topologies (e.g. background `threading.Thread` + `schedule`, or `apscheduler.BlockingScheduler`): all ruled out by the asyncio-loop integration; document the choice.

5. **Q11 from `to-architect-snapshot-2026-06-01.md:966`:** your "ADR-006 vs ADR-007 split" answer was: **ADR-007 = scheduler side-effect & lifecycle (this wave); ADR-006 = in-memory/runtime-state choices (separate, not blocking B6)**. ADR-006 is on your backlog. The B6 scope is fully covered by this packet.
